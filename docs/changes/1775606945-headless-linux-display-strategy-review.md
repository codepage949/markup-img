# Linux 헤드리스 디스플레이 전략 검토

## 개요

`xvfb`와 같은 기능을 프로젝트 내부에서 직접 구현해 사용할 수 있는지 검토한다.

## 현재 전제

- 현재 렌더링은 Neutralinojs의 Linux 네이티브 웹뷰(GTK/WebKit 계열)에 의존한다.
- 이 경로는 실제 디스플레이 서버(X11/Wayland) 연결이 없으면 `GtkStyleContext` 생성 단계에서 실패한다.
- `xvfb-run`은 애플리케이션이 요구하는 디스플레이 서버를 바깥에서 가상으로 제공하는 역할을 한다.

## 검토 결과

### 1. `xvfb` 자체를 앱 내부에서 구현하는 것은 사실상 비현실적

- `xvfb`는 단순 유틸리티가 아니라 X11 서버 구현체다.
- 이를 대체하려면 최소한 가상 framebuffer, 윈도우 관리, 이벤트 처리, GTK/WebKit이 기대하는 X11 프로토콜 대응이 필요하다.
- 즉 현재 프로젝트 범위를 크게 넘어서는 별도 시스템 소프트웨어를 만드는 수준이다.

### 2. "비슷한 기능"을 내부적으로 감싸는 것은 가능

- 앱이 `DISPLAY` 유무를 감지한 뒤 `Xvfb`를 자식 프로세스로 직접 띄우고,
  그 환경변수(`DISPLAY`)를 설정한 상태에서 Neutralino를 실행하는 방식은 가능하다.
- 이 경우 사용자는 `xvfb-run` 명령을 직접 치지 않아도 된다.
- 다만 본질적으로는 `xvfb`를 "직접 구현"하는 것이 아니라 시스템의 `Xvfb` 바이너리를 래핑하는 것이다.

### 3. 완전한 자체 내장 방식은 다른 렌더링 백엔드 교체에 가까움

- 시스템 디스플레이 서버 없이 동작하려면 Neutralino/Linux 웹뷰 경로를 버리고,
  헤드리스 렌더링이 가능한 다른 엔진으로 갈아타야 한다.
- 예:
  - headless Chromium/Playwright/Puppeteer
  - 별도 offscreen WebKit/WPE 기반 런타임
  - 서버 렌더링 또는 DOM 캡처 방식 자체 변경
- 이건 `xvfb` 대체 구현이 아니라 아키텍처 변경이다.

## 현실적인 선택지

### 선택지 A. `Xvfb`를 의존성으로 두고 자동 실행

- 장점:
  - 현재 Neutralino 구조를 거의 유지 가능
  - 구현 난이도가 가장 낮음
  - 사용자 경험상 `xvfb-run` 수동 입력 제거 가능
- 단점:
  - Linux 시스템에 `Xvfb`가 설치돼 있어야 함
  - 순수 단일 바이너리는 아님

### 선택지 B. `Xvfb` 바이너리를 함께 배포하고 내부에서 실행

- 장점:
  - 사용자 시스템 의존성 축소
  - CLI 사용성 개선
- 단점:
  - 배포 크기 증가
  - 플랫폼/배포판 호환성 관리 필요
  - 라이선스 및 패키징 관리 필요

### 선택지 C. 렌더링 엔진을 헤드리스 친화적인 것으로 교체

- 장점:
  - Linux 디스플레이 서버 의존 자체 제거 가능
  - CI/서버 환경 호환성 향상
- 단점:
  - Neutralino 기반 초경량 구조와 현재 구현 장점 일부 상실 가능
  - 마이그레이션 비용 큼

## 결론

- `xvfb`와 같은 기능을 프로젝트 내부에서 "직접 구현"하는 것은 현실적으로 권장되지 않는다.
- 현재 구조를 유지하면서 사용자 경험만 개선하려면 `Xvfb`를 자동으로 실행하는 래퍼 전략이 가장 현실적이다.
- 디스플레이 서버 의존을 완전히 없애고 싶다면, 그건 `xvfb` 재구현이 아니라 렌더링 백엔드 교체 문제로 봐야 한다.

## 권장 방향

1. 단기: Linux에서 `DISPLAY`가 없으면 `Xvfb`를 자동 기동하는 방식 검토
2. 중기: `Xvfb` 미설치 시 명확한 안내 메시지 제공
3. 장기: 서버/CI 헤드리스 사용 비중이 크다면 렌더링 백엔드 재검토
