# GitHub Pages 랜딩 페이지 추가

## 배경

- 저장소는 CLI 중심 문서만 제공하고 있어 GitHub Pages용 첫 화면이 없다.
- 릴리즈 다운로드와 저장소 진입점을 한 번에 안내할 랜딩 페이지가 필요하다.
- 최신 버전은 수동으로 적기보다 릴리즈 정보에 맞춰 자동으로 표시되는 편이 유지보수에 유리하다.

## 목표

- `github.io`에서 사용할 단일 랜딩 페이지를 추가한다.
- GitHub 저장소 버튼과 릴리즈 다운로드 버튼을 배치한다.
- 최신 릴리즈 버전을 동적으로 표시하고, 실패 시에도 페이지가 자연스럽게 동작하도록 한다.

## 구현 계획

1. GitHub Pages 진입 파일을 `docs/index.html`로 추가한다.
2. GitHub API로 최신 릴리즈를 조회해 버전 표시와 다운로드 링크를 갱신한다.
3. 정적 HTML/CSS/JS만으로 반응형 레이아웃을 구성한다.

## 검증 계획

- HTML 구조와 스크립트 동작을 직접 점검한다.
- `docs/index.html`을 로컬에서 열어 주요 문구와 링크, 반응형 레이아웃을 확인한다.

## 구현 내용

- `docs/index.html`을 추가해 제품 소개, 주요 특징, 사용 흐름, CTA 영역을 포함한 랜딩 페이지를 구성했다.
- 히어로 헤드라인은 타이핑 애니메이션으로 순환 문구가 바뀌도록 구성했고, 문구 길이나 줄바꿈에 따라 카드 레이아웃이 흔들리지 않도록 고정 높이와 기준 문구를 함께 사용했다.
- GitHub 버튼은 저장소 메인으로, 다운로드 버튼은 최신 릴리즈 페이지로 연결되도록 기본 링크를 설정했다.
- 브라우저에서 GitHub Releases API를 호출해 최신 태그와 발행일을 반영하고, 조회가 성공하면 다운로드 버튼도 최신 릴리즈 URL로 교체하도록 구현했다.
- API 호출 실패 시에는 `Cargo.toml`의 현재 버전을 기본값으로 보여 주고, 버튼은 일반 릴리즈 목록으로 유지하도록 폴백을 넣었다.

## 검증 결과

- `docs/index.html`의 정적 구조를 점검했다.
- 로컬 HTTP 서버로 문서를 열어 최신 버전 표시와 버튼 링크 갱신 로직을 확인했다.

## 다음 판단

- Pages 배포는 저장소 설정에서 배포 소스를 `docs/`로 지정하면 바로 사용할 수 있다.
- 추후 운영체제별 직접 다운로드 버튼이 필요하면 릴리즈 assets를 읽어 플랫폼 선택 UI를 추가하는 편이 자연스럽다.
